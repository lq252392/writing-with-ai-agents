# 章节规划

## Overview

把"这一章想写什么"压成可执行的章节计划。重点是确定本章必须推进什么，而不是提前写正文。

## Trigger Keywords

- 规划这一章
- 拆场景
- 本章怎么分
- 给我章节计划
- 先定停点

## State Contract: Inputs

必读字段：

- `next_stage_entry`
- `carry_over_risks`
- `confirmed_constraints_touched`
- `state_version`
- `constraints_version`

可选字段：

- `scene_summary`
- `character_state_updates`

忽略字段：

- 与本章无关的远端剧情设想
- 未确认的自由扩展脑洞

## When to Use

适用于：

- 有本章目标，但还没拆成场景
- 容易写着写着偏离本章任务
- 角色多、信息多，怕顺序写乱

不适用于：

- 只有模糊创意，先做头脑风暴
- 已有足够细的场景清单，直接扩写更好

## Planning Checklist

规划时至少明确：

- 本章开始状态
- 本章结束状态
- 本章主推进点
- 本章不能越过的边界
- 场景序列：每场戏只负责一个推进动作
- 哪些信息本章只能埋，不该揭

如果是多人场景或规则型场景，再补两项：

- 每场戏有哪些在场角色，谁必须持续在场
- 每场戏结束后，谁的认知、关系或局势发生了什么变化

## Output

输出建议包括：

- 章节目标
- 场景列表
- 每场戏的推进职责
- 角色状态变化
- 结尾停点
- 每场 `target_word_range`

为了让 `novel-scene-drafting` 更稳，场景列表最好至少写清：

- 场景入口状态
- 在场角色
- 本场只推进的一个主动作
- 本场允许揭示的信息上限
- 场景出口状态
- 本场 `target_word_range`

## Red Flags - STOP and Start Over

- "我直接写一万字的正文大纲吧（越俎代庖）"
- "计划太麻烦了，我只写个大概方向然后开始写正文"
- "场景里堆了三四个主要矛盾，主角一边破案一边谈恋爱一边发现身世"
- "既然是计划，我不用管上下章的连贯性"

**如果出现以上情况：立即停止，恢复到只做"推进点拆解"和"边界限制"。**

## Rules

- 不要把章节计划写成正文。
- 一章只围绕少数关键推进点，不并列堆太多支线。
- 场景顺序要能形成因果，不要只是事件列表。
- 规划前先读取最新状态摘要和长期约束；若计划与已确认事实冲突，先停下修计划，不要把矛盾留给扩写阶段硬扛。
- 计划里要显式保留多人在场信息，不能默认"写的时候自然会带出来"。
- `target_word_range` 要给区间，不要给单点死数；范围只服务于控制篇幅，不替代场景职责。
